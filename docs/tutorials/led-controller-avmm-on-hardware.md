# Programming the Board: Where IPCraft's Pipeline Ends

Part 5 of the [LED Controller on Real Hardware](led-controller-avmm-authoring.md)
series — the payoff. Everything up to here (`.ip.yml`/`.mm.yml` → generated
RTL → cocotb simulation → import cross-check → headless Quartus timing
report) is real, tested, and automatable. This part is different, and says
so plainly: **IPCraft has no bitstream-generation or board-programming
step.** From here, every step is cvsoc's own existing manual/scripted flow —
`quartus_pgm`, `nios2-download`, `nios2-terminal` — run by a human with a
physical [Terasic DE10-Nano](https://www.terasic.com.tw/) and a USB-Blaster
in hand.

!!! info "What you will learn"
    - Exactly where IPCraft's automated pipeline stops, and why that's an
      honest, useful thing to document rather than paper over.
    - How to close the loop from a generated register file to observable
      behavior on real Nios II software.
    - What was actually verified in this session versus what requires
      physical hardware access this environment doesn't have.

## The full round-trip, in software

`software/app/main.c` extends `04_nios2_led`'s pattern. A hand-authored
Platform Designer component gets no auto-generated HAL macro header (unlike
the stock `altera_avalon_pio`), so registers are poked directly by byte
offset, matching `led_controller_avmm.mm.yml`:

```c
/* Startup self-test: VERSION must match what IPCraft generated. */
uint32_t version = IORD_32DIRECT(LED_CTRL_BASE, 0x00);
/* ... cycles LED_PATTERN through the same demo patterns as 04_nios2_led ... */
uint32_t events = IORD_32DIRECT(LED_CTRL_BASE, 0x08);
if (events & HEARTBEAT_TOGGLED) {
    printf("heartbeat\n");
    IOWR_32DIRECT(LED_CTRL_BASE, 0x08, HEARTBEAT_TOGGLED); /* W1C */
}
```

This closes the full loop the series set out to build: a register bank
described in YAML, generated by IPCraft, simulated, synthesized — and now
read and written by real embedded C, with a startup self-test and a
write-1-to-clear round trip through live hardware.

## What this session actually verified

Full transparency on what was and wasn't checked, since a physical board
isn't available in this environment:

| Step | Verified how |
|------|--------------|
| `main.c` compiles cleanly | Real `nios2-elf-gcc` cross-compiler (Quartus 25.1std's Nios II EDS, on the same remote workstation used for Parts 1 and 4), `-Wall -Wextra`, zero warnings |
| Register offsets match the generated register file | Cross-checked against `led_controller_avmm.mm.yml` and the cocotb-verified behavior from Part 2 |
| The peripheral itself works | GHDL/cocotb simulation, Part 2 — 4/4 real assertions passing |
| Timing closes at 50 MHz | Real Quartus 25.1std synthesis, Part 4 — positive slack on all corners |
| **LEDs actually blink; `heartbeat` actually prints** | **Not verified here** — requires a physical DE10-Nano + USB-Blaster, which this environment does not have |

## Programming the board (for whoever has the hardware)

```bash
cd 16_ipcraft_led_avmm/quartus
make all                        # qsys → Quartus compile → BSP → app
make program-sof                # JTAG-program the .sof (SRAM, not persistent)
make download-elf                # download + run the Nios II ELF
make terminal                    # JTAG UART — watch for "heartbeat"
```

Expected: the same LED cycling pattern as `04_nios2_led`, now driven by an
IPCraft-generated register file; `led_controller_avmm VERSION OK:
0x00000100` at startup; `heartbeat` printed roughly once per second. If the
write-1-to-clear didn't actually work in real hardware, the print would
either stop appearing correctly or the polling loop would misbehave — a
behavioral proof, not just a visual LED check.

## Closing the series

Across five parts, this series built one real peripheral — the exact one
cvsoc's own roadmap called for and never built — entirely through IPCraft,
and in doing so found and fixed **six confirmed bugs** across the generator:

1. An Avalon-MM slave with no `useOptionalPorts` produced invalid VHDL
   (empty array truthiness in Nunjucks).
2. `mm_loader.py.j2` read snake_case keys against a camelCase-only schema —
   every generated cocotb test's register list was silently empty.
3. `_parse_bits` never stripped the schema's `[`/`]` brackets.
4. The generated Avalon-MM cocotb helpers shifted addresses by `>> 2`,
   mismatched against the generator's own byte-addressed register file.
5. The generated `_read_reg` sampled one cycle too early for a
   fixed-latency slave.
6. `quartus_sdc.j2`'s clock constraint was silently swallowed into a
   Tcl comment; `quartus_project.tcl.j2`'s `SDC_FILE` path pointed one
   directory too high once that was fixed.

Every one of these was invisible in `ipcraft-spec`'s own CI-exercised
examples — because a synthetic fixture never has to actually work, and
nothing there ever asserted, ran headlessly on real Quartus, or was cross-
checked against real, hand-written production HDL. A real target does. That
was the whole premise of this series, and it held.
